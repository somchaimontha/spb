01 สถานการณ์ปัจจุบันและปัญหา

จากการวิเคราะห์ไฟล์ที่ใช้งานจริง พบว่าปัจจุบันงานแผนและงบประมาณของวิทยาลัย ดำเนินการผ่านไฟล์ Excel จำนวนหลายไฟล์ ซึ่งทำงานแยกจากกัน:

ไฟล์หน้าที่ปัจจุบันประเด็น
budget 2565 ของนวมินทร์.xlsx ต้นแบบ ระบบบัญชีงบประมาณครบวงจร 108 ชีต — Note · งปม.เข้า · จัดสรรงปม. · รายการจัดสรร + ชีตทะเบียนรายจ่ายรายรหัส (101–920) แต่ละชีตเก็บรายการเบิก 30+ บรรทัด คนเดียวเปิดได้ทีละคน, สูตรอาจพังเมื่อแก้, ไม่มีประวัติอนุมัติ
รายชื่อโครงการใหม่ 2569.xlsx เก็บโครงการ ~140 โครงการ แยกตามฝ่าย/แผนก พร้อมแหล่งงบ ติดตามสถานะอนุมัติยาก ไม่ผูกกับการเบิกจริง
budgetControl.xlsx บัญชีคุมงบเวอร์ชันถัดมา — ใช้แนวเดียวกับ budget 2565 หน้าตาเดียวกัน เสี่ยงคีย์ผิด
สรุป-คชจ 2569.xlsx สรุปจัดสรร/ใช้จริง/คงเหลือรายไตรมาส แยกแหล่งงบ ต้องคีย์ซ้ำจาก budgetControl ทุกครั้ง
การบริหารงบประมาณ.pptx สไลด์ประกอบการประชุมงบประมาณ ข้อมูลล้าสมัยทันทีหลังประชุม

โครงสร้างไฟล์ต้นแบบ (budget 2565)

ผมวิเคราะห์ไฟล์นี้แล้ว แบ่งชีตเป็น 5 กลุ่มหลัก ทั้งหมดจะถูกย้ายมาเป็นเว็บแอปแบบ 1:1 :

กลุ่มชีตเนื้อหาเทียบในระบบใหม่
Noteชื่อสถานศึกษา + ปีงบประมาณหน้า Settings
งปม.เข้าทะเบียนรับเงิน — วันที่, ชื่อเอกสารจัดสรร, รหัสเอกสาร, แยก 14 ประเภทเงินโมดูล “รับงบ”
จัดสรร งปม.Master sheet จัดสรรงบไปแต่ละรหัส 101–920 พร้อม % ใช้ไป + คงเหลือหน้า Dashboard + บัญชีคุมงบ
รายการจัดสรรสรุปรายการ + แยกตามเดือน (พ.ย.–ก.ย.)รายงานรายเดือน
101 — 920100 ชีต — 1 ชีตต่อ 1 รหัสรายจ่าย เก็บทะเบียนเบิก รหัส/วันที่/ผู้ขอใช้/แผนก/ยอดแยก 14 ประเภท/รวมโมดูล “ใบขอเบิก”

ปัญหาหลัก

  • เปิดพร้อมกันไม่ได้ — ไฟล์ Excel ในเครื่อง, คนคีย์ต้องรอกัน
  • 108 ชีตหน้าตาเดียวกัน — เสี่ยงคีย์ผิดช่อง สูตรหลุด ลำบากตรวจ
  • ผู้ขอใช้งบไม่เห็นยอดคงเหลือ — ต้องโทรถาม/เดินไปถามงานแผนทุกครั้ง
  • การอนุมัติยังเป็นกระดาษ — ไม่มี audit log ค้นย้อนยาก
  • รายงานสำหรับผู้บริหารทำมือ — ใช้เวลามาก ตัวเลขอาจล้าสมัย

02 วัตถุประสงค์และตัวชี้วัด

วัตถุประสงค์

  • รวมทะเบียนโครงการ + บัญชีคุมงบ + ใบขอเบิก เป็นระบบเดียว
  • ผู้รับผิดชอบโครงการเห็นยอดคงเหลือของตนเอง real-time
  • ผู้บริหารดูภาพรวมการใช้งบประมาณรายเดือน/รายไตรมาสได้ทันที
  • เก็บไฟล์เอกสารประกอบ (โครงการ, ใบเสร็จ) ไว้ใน Drive โยงกลับมาที่รายการเบิก

ตัวชี้วัด

  • เวลาทำรายงานสรุปงบประมาณรายเดือน < 5 นาที (จากเดิม >1 วัน)
  • ผู้รับผิดชอบโครงการเข้าใช้ระบบ > 80% ในไตรมาสแรก
  • ใบขอเบิก 100% ผ่านระบบภายในปีงบฯ 2570
  • ความผิดพลาดยอดคงเหลือ = 0 (มีการ sync จาก Sheet หลักเสมอ)

04 ขอบเขตและโมดูล

ระบบประกอบด้วย 6 โมดูลหลัก แยกตามภารกิจของงานแผน:

M1 · ทะเบียนโครงการ (Plan)
staff · head · admin
  • เพิ่ม/แก้ไขโครงการประจำปี อ้างอิงตามฟอร์มเดิม
  • ระบุแหล่งงบ: เรียนฟรี / บกศ. / รายจ่ายอื่น / ส่วนกลาง สอศ.
  • ผูกกับฝ่าย+แผนก+ผู้รับผิดชอบ
  • สถานะ: ร่าง · เสนอ · อนุมัติ · ยกเลิก
M2 · บัญชีคุมงบ (Budget Ledger)
admin
  • หมวดเงิน 101–414 (4 หมวดหลัก, 30+ รายการย่อย)
  • × 13 ช่องประเภทเงิน (ปวช./ปวส./ระยะสั้น/อุดหนุน/บกศ. ฯลฯ)
  • คำนวณ ได้รับ − ใช้ไป → คงเหลือ อัตโนมัติ
M3 · ใบขอเบิก (Disbursement)
staff → head → admin
  • ฟอร์มขอเบิกจากโครงการที่ได้รับอนุมัติแล้ว
  • แนบไฟล์ใบเสร็จ/รายงาน ใน Drive
  • มี workflow อนุมัติ + ประวัติการกระทำ
  • ออกเลขที่ชุดเบิกอัตโนมัติ
M4 · Dashboard
ทุกบทบาท
  • ยอดได้รับ · ใช้ไป · คงเหลือ ทั้งวิทยาลัย
  • กราฟใช้จ่ายรายไตรมาส แยกแหล่งงบ
  • Top 10 โครงการที่ใช้งบมากสุด
  • Alert: หมวดที่งบใกล้หมด (<10%)
M5 · รายงาน (Reports)
admin · executive
  • สรุปรายไตรมาส (ตามแบบ “สรุป-คชจ”)
  • รายงานรายโครงการ + Export PDF
  • Export Excel ในรูปแบบเดิมเพื่อส่งต่อ สอศ.
M6 · ตั้งค่า (Settings)
admin
  • เปิดปีงบประมาณใหม่ (clone หมวด)
  • จัดการผู้ใช้ + บทบาท
  • ตั้งค่าฝ่าย/แผนก/หมวดเงิน

05 สถาปัตยกรรมระบบ

ใช้รูปแบบ Sheets-as-Database: ผู้ใช้เข้าผ่านเว็บ Static (GitHub Pages) เรียก API ที่เขียนด้วย Google Apps Script — โดย Apps Script จะติดต่อ Sheets และ Drive ในโดเมนของวิทยาลัยโดยตรง ข้อดีคือ ไม่มี Server ต้องดูแล, ฟรี, และเจ้าหน้าที่ยังเปิด Sheet ขึ้นมาตรวจ/แก้ตอนฉุกเฉินได้

CLIENT (BROWSER)
Static Web App
  • HTML + CSS + Alpine.js
  • Chart.js สำหรับกราฟ
  • Tailwind CSS via CDN
  • Hosted: GitHub Pages
API LAYER
Apps Script Web App
  • doGet / doPost
  • Verify Google ID Token
  • RBAC ตามตาราง Users
  • Endpoint: /exec?action=…
DATA
Google Workspace
  • Spreadsheet หลัก 1 ไฟล์
  • Drive folder ตามปีงบฯ
  • เก็บประวัติด้วย Sheet Logs
↓   HTTPS · Google ID Token   ↓
การยืนยันตัวตน: Google Identity Services (One Tap) — รับเฉพาะอีเมลโดเมนวิทยาลัย
การจำกัดสิทธิ์: Apps Script ตรวจ email ใน Sheet Users ทุก request
ทำไมไม่ใช้ Cloud Functions / Firebase? เพราะเจ้าหน้าที่งานแผนต้องสามารถเปิด Sheet ได้ตามปกติเพื่อตรวจ/แก้ หากย้ายไปฐานข้อมูลอื่นจะกลายเป็น “กล่องดำ” ที่ต้องพึ่งโปรแกรมเมอร์ทุกครั้ง

ตัวอย่าง doPost

function doPost(e) {
  const body = JSON.parse(e.postData.contents);
  const user = verifyToken(body.id_token);     // ตรวจกับ Google
  if (!user) return json({ ok: false, err: 'unauthorized' });

  const me = lookupUser(user.email);                // อ่าน Sheet Users
  if (!me || !me.active) return json({ ok: false, err: 'forbidden' });

  const handler = ROUTES[body.action];
  if (!handler) return json({ ok: false, err: 'unknown_action' });

  if (!can(me, body.action)) return json({ ok: false, err: 'forbidden' });

  log_(me.email, body.action, body.payload);
  return json(handler(me, body.payload));
}

11 การติดตั้งและ Deploy

  1. เตรียม Spreadsheet — copy จาก template PlanWork_DB.template เปลี่ยนชื่อ + ใส่ปีงบฯ ปัจจุบัน
  2. เตรียม Drive folder — โครงสร้าง /PlanWork/{fy}/{project_id}/
  3. สร้าง Apps Script project — ผูกกับ Spreadsheet (Container-bound)
  4. clasp push โค้ดจาก apps-script/ — เปิด Web App → Deploy → New deployment → Anyone
  5. ใส่ Web App URL ใน api.js ของ frontend
  6. Push frontend ไป GitHub — เปิด GitHub Pages branch main /docs หรือ gh-pages
  7. ตั้ง Custom domain (ถ้ามี): plan.msc.ac.th
  8. ทดสอบกับผู้ใช้นำร่อง 2-3 คนใน 1 ฝ่าย ก่อนเปิดทั้งวิทยาลัย

12 ความปลอดภัยและสำรองข้อมูล

  • Auth: Google ID Token + ตรวจโดเมน (เช่น @msc.ac.th)
  • RBAC: ตรวจในทุก action โดยอ้างอิง Sheet Users.role
  • Audit: ทุกการสร้าง/แก้/ลบ — เขียน SystemLogs และ ApprovalLogs
  • Backup: Apps Script trigger รายวัน — copy Spreadsheet เก็บใน folder /PlanWork/_backup/{yyyy-mm-dd}.xlsx
  • Version control: Sheet ใช้ Version history ของ Google ได้อยู่แล้ว
  • Drive: share folder ระดับ “เฉพาะคนในโดเมน” ไฟล์โครงการ inherit สิทธิ์

14 ความเสี่ยงและทางเลือก

ความเสี่ยงผลกระทบการลดความเสี่ยง
ขีดจำกัด Apps Script (6 นาที/run, 30 trigger/วัน) รายงานใหญ่อาจ timeout ทำเป็น batch, cache, สร้าง PDF แบบ async
